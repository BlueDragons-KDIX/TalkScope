その他気になる事項

調査レポートの主要分類には入れにくいが、統合作業で気にしたい点をまとめる。

運用・設定

  • THEME_EMA_ENABLED がコード定数のため、本番・検証で切り替えるにはコード変更が必要。 環境変数化するか、MVP中は無効固定と明記するか決めたい。
  • TERM_IDF_LOAD_MIN_VALUE と IDF_JSON_PATH は .env.example にあるが、 Backend/AGENTS.md の「.envを読まない」と衝突している。
  • IDF辞書データが同梱されていないため、IDFバフの検証はテスト用JSONかDB投入データに依存する。

API設計

  • /analysis/score/terms は term_vector を必須入力にしている。 フロントがこのベクトルをどう得るのか、既存 /analysis/vectorize とどうつなぐのかを決める必要がある。
  • TermScoreInput は term_vector に min_length がない。 空配列でもバリデーションは通り、テーマ類似バフが付かないだけになる。
  • occurrence_count は省略可能。 フロント側で出現回数を計算するのか、バックエンドでテキストから計算するのかが未確定に見える。

DB設計

  • 既存の dictionary_terms.idf_wiki は Integer だが、IDFは通常小数を使うため Float の term_idf.idf_value のほうが自然。
  • TermIdf は app.models.dictionary.Base にぶら下がっている。 create_all対象にはなるが、辞書DBモデルとの境界は曖昧。
  • term_idf 用のCRUDや投入スクリプトがない。 ドキュメントではバッチ投入想定だが、実装はまだ受け皿だけ。

実装品質

  • score_building_blocks.py は将来用の関数も含んでおり、今のMVPに対してはやや大きい。 残すなら「公開APIで使う範囲」と「実験・将来用」をコメントで分けたい。
  • term_score.py はセッション管理、形態素フィルタ、API応答組み立てを一箇所に持つ。 現状は許容範囲だが、テーマ永続化やDB連携が増えるなら分割候補。
  • main.py はAGENTS上はアプリ生成とrouter includeに寄せたいが、現在はdotenv、DB初期化、IDFロード方針も入っている。

ドキュメント

  • Backend/docs/IDF_DATA_PIPELINE.md は「Wikipedia由来IDFがある前提」に読めるが、実データは未同梱。 読み手に誤解が出やすい。
  • Backend/docs/SCORE_ANALYSIS_API.md と Backend/README.md は、新規APIを説明している。 実装側でIDFロードがコメントアウトされている状態と同期が必要。
  • ルートREADMEはハッカソン提出テンプレート維持のルールがあるため、スコア計算の詳細はBackend docs側に閉じたほうがよい。

確認したい小さな違和感

  • Backend/main.py の init_idf_table import は、呼び出しを止めるなら未使用になる。
  • idf_runtime.py はDB接続エラーを握って None にする方針だが、import不整合だけは握れない。
  • get_theme() は THEME_EMA_ENABLED=False のとき保存済みセッションを読まない。 テスト中に set_theme() しても無効化されるのは意図通りだが、初見では少し驚く。
