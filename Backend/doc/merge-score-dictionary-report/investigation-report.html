merge-score-dictionary 調査レポート

スコア計算ロジックのマージ後に確認した矛盾点、未使用箇所、名前・内容の重複、テスト方法の違いをまとめる。 本レポートは静的調査ベースであり、コード修正はまだ行っていない。

要約

分類内容影響
矛盾点 Backend/main.py は init_idf_table を import しているが、lifespan 内の呼び出しはコメントアウトされている。 中: IDF バフは常に無効に近い状態になる。
矛盾点 Backend/app/services/idf_runtime.py は from app.core.database import db を前提にしているが、現在のDB層は get_database() / get_transaction_manager() 中心。 高: import 時点で落ちる可能性がある。
矛盾点 Backend/AGENTS.md には「.envを読まない」とある一方、Backend/main.py は Backend/.env を読み込む。 中: 運用ルールと実装方針が不一致。
未使用箇所 score_building_blocks.py の PMI / bigram 系関数はHTTP経路では利用されない。 低: テスト済み補助関数だが、MVPに必要か判断が必要。
名前の重複 dictionary_terms.idf_wiki と新規 term_idf.idf_value が似た概念を別テーブルで持つ。 中: どちらを正とするか曖昧。
内容の重複 既存 text_analysis.tfidf_scores() と新規IDFバフ設計が、どちらもIDF/Tf-idf語スコアに関わる。 低: 目的は違うが、説明なしでは混同しやすい。
テスト方法の違い スコア系テストは sys.path を個別に挿入するものが多く、既存テストは from main import app だけのものもある。 低: 実行場所により成功条件が変わりやすい。

矛盾点

  • Backend/main.py で init_idf_table を import しているが、実行行はコメントアウトされている。 そのため term_score.compute_term_scores_for_request() が get_idf_table() を呼んでも、通常は None になる。
  • idf_runtime.py は app.core.database.db というグローバル変数を参照するが、現在の database.py にはその名前がない。 現在の設計は get_database() で共有インスタンスを返す方式。
  • Backend/docs/IDF_DATA_PIPELINE.md では起動時にDB初期化後 init_idf_table() を実行すると書かれているが、 現在の main.py ではコメントアウトされている。
  • Backend/AGENTS.md は .env を読まない方針だが、main.py は load_dotenv() で Backend/.env を読む。

利用していない箇所

  • term_score.adjacent_bigrams_for_scoring_text() はテスト用・補助用であり、 POST /analysis/score/terms からは呼ばれない。
  • score_building_blocks.window_bigram_counts(), log_pmi_with_smoothing(), marginal_counts_from_bigrams(), term_adjacent_ppmi_max() は現行HTTP経路では未使用。
  • score_building_blocks.best_sense_index_by_context() はmulti-sense将来対応向けで、現行HTTP経路では未使用。
  • Backend/.env.example に IDF_JSON_PATH=data/idf/ja_wiki_idf.json の例があるが、該当JSON本体はリポジトリ内に見当たらない。

名前の重複・概念の衝突

  • 既存の dictionary_terms.idf_wiki と、新規の term_idf.idf_value がどちらもWikipedia IDF風の値を表せる。 ただし型は前者が Integer、後者が Float。
  • term と lemma の使い分けが要注意。スコア計算は lemma を前提にしているが、辞書DBは term を主キー的に扱う。
  • score という語が、既存の refer_dictionary_get_scores 系と新規 /analysis/score/terms 系で別意味になっている。

内容の重複

  • text_analysis.tfidf_scores() は入力コーパス内でTF-IDFを計算する。 新規 term_idf / IdfLookupTable は外部コーパス由来の語ごとのIDFマスタを読む。 目的は違うが、どちらも「IDF」を扱うためREADMEでの整理が必要。
  • 形態素フィルタが text_analysis._terms_for_tfidf() と term_score.morph_tokens_for_scoring() に分かれている。 前者は名詞・動詞・形容詞・ALPHAを通し、後者は名詞・固有名詞中心で代名詞を除外する。
  • テーマベクトル、用語ベクトル、意味ベクトルがすべてベクトルとして登場する。 dictionary.meaning_vector は300次元固定、term_vector はAPI入力で次元制約が弱い。

テスト方法の違い

  • スコア系テストは sys.path.insert を各ファイルで実施している。 一方で既存APIテストは from main import app のみのものがある。
  • test_idf_runtime.py は一時JSONを作成してローダを単体テストする。 DB読み込みは _load_idf_lookup_from_database を monkeypatch するため、実DBとの結合確認は薄い。
  • test_score_analysis_endpoint.py は TestClient(app) をモジュールグローバルで作成する。 lifespanとIDF初期化の挙動を検証するには、別途lifespanを明示的に踏むテストが必要。
  • 既存辞書DB系には integration marker やDB availabilityによるskipがある。 スコア系のIDF DBロードは同じ粒度のintegrationテストがまだない。

優先して確認すべきこと

  1. idf_runtime.py のDB参照方式を現在の get_database() 方式に合わせるか判断する。
  2. init_idf_table() を本当に起動時に呼ぶのか、IDF辞書がない間はimportも外すのか判断する。
  3. dictionary_terms.idf_wiki と term_idf の責務を決める。
  4. スコア計算APIをMVPに含める場合、term_vector を誰が生成して渡すのかを決める。
  5. テスト実行方法を cd Backend && uv run pytest に寄せるか、各テストの sys.path を整理する。
