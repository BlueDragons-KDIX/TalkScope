今後のワークフロー

スコア計算ロジックと辞書DBまわりを安全に統合するための作業順。 いきなり修正せず、調査・判断・最小修正・検証の順で進める。

Phase 1: 現状固定と問題分類

  1. git status --short で未解決・未追跡・意図しない差分を確認する。
  2. Backend/main.py の衝突解消後の状態を確認する。
  3. DB接続方式、IDFロード方式、API登録状態、テスト追加分を分類する。
  4. 本ディレクトリの調査HTMLを更新する。

Phase 2: 方針決定

  1. IDFバフを今回有効にするか、将来用として無効のまま残すか決める。
  2. term_idf と dictionary_terms.idf_wiki の正を決める。
  3. term_vector の生成元を決める。
  4. THEME_EMA_ENABLED の切り替え方法をコード定数のままにするか決める。
  5. .env 読み込み方針をAGENTSのルールと合わせる。

Phase 3: 最小修正候補

  1. idf_runtime.py の db 参照を現在のDB APIに合わせる。 例: get_database() を使う。
  2. main.py の init_idf_table を呼ぶか、呼ばないならimportとドキュメントを整理する。
  3. term_idf の実データがない前提で、IDFなしでもAPIが落ちないことを確認する。
  4. 未使用の将来用関数は残す場合、ドキュメントに「HTTP未使用」と明記する。

Phase 4: テスト整理

  1. cd Backend && uv run pytest tests/test_score_building_blocks.py
  2. cd Backend && uv run pytest tests/test_term_score.py
  3. cd Backend && uv run pytest tests/test_idf_runtime.py
  4. cd Backend && uv run pytest tests/test_score_analysis_endpoint.py
  5. 辞書系との結合確認として tests/test_dictionary_*.py と tests/test_refer_dictionary_endpoint.py を必要範囲で実行する。

DBが必要なテストは integration marker やDB availability skipを前提に扱う。 依存同期が必要な場合は uv sync がインストール系コマンドに該当するため、事前確認を取る。

Phase 5: README / docs 同期

  1. Backend/README.md のスコアAPI説明を実装状態に合わせる。
  2. Backend/docs/IDF_DATA_PIPELINE.md に、IDF実データが未同梱であることを書くか判断する。
  3. Backend/docs/SCORE_ANALYSIS_API.md に、テーマEMAがデフォルト無効である点を明確にする。
  4. ルートREADMEへの影響があるなら提出テンプレートを崩さない形で同期する。

完了条件

  • アプリ起動時に idf_runtime のimportで落ちない。
  • IDFデータがなくても /analysis/score/terms が正常に動く。
  • IDFデータを入れた場合の読み込み経路が1つに説明できる。
  • 辞書DBの idf_wiki とスコアDBの term_idf の関係が説明できる。
  • スコア系・辞書系の主要テストの実行方法がREADMEまたはdocsに揃っている。
