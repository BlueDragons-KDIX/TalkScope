調査中の思考メモ

マージ後の違和感や確認順を小まめに残すためのメモ。 後で正式な修正方針に落とす前の、観察ログとして使う。

2026-05-23 調査開始

最初に見るべきは Backend/main.py。 マージ衝突の中心で、DB初期化・dotenv・router・IDF初期化が集まっている。

main.py は init_idf_table を import しているが、実行はコメントアウト。 これは「今は壊したくないので止めた」状態に見える。 ただし未使用importになり、さらに idf_runtime.py のimport自体が壊れるならアプリ起動に影響する。

idf_runtime.py が from app.core.database import db を使っている。 現在の database.py は get_database() を提供する構成なので、ここはマージ時に取り残された可能性が高い。

IDF辞書の実データは見つからない。 .env.example の IDF_JSON_PATH=data/idf/ja_wiki_idf.json は例だけで、ファイル本体はない。 したがって、今すぐIDFバフを有効化する運用はまだ成立していない。

既存辞書DBには dictionary_terms.idf_wiki がある。 新規スコア側には term_idf.idf_value がある。 名前の意味は近いが、保存先・型・用途が違う。ここはチーム内で合意が必要。

スコア計算APIは term_vector をクライアント入力として受け取る。 既存辞書の meaning_vector を使うのか、フロント側で別途ベクトルを作るのか、現時点では接続が見えにくい。

THEME_EMA_ENABLED はコード定数で False。 テーマEMAのAPIは生えているが、デフォルトではテーマ更新もテーマ類似バフも無効。 使うにはコード変更が必要という設計。

score_building_blocks.py は機能が多い。 現行APIに直結するのは cosine / EMA / IDF / count axis / compose あたり。 PMIやmulti-sense選択は将来用に見える。

テストはよく書かれているが、スコア系だけ sys.path をテストファイルごとに差し込む傾向がある。 既存テストの流儀と少し違う。実行ディレクトリ依存を減らすなら、pytest設定側に寄せたほうがよさそう。

次に深掘りしたい問い

  • IDFバフは今回のMVPで必要か、それとも将来用として寝かせるか。
  • term_vector の生成元は辞書DBの meaning_vector か、解析APIの出力か、フロント入力か。
  • テーマEMAをコード定数ではなく環境変数や設定にするべきか。
  • idf_wiki を廃止予定にするか、term_idf へ統合するか。
  • 起動時DB初期化とIDFロードを同じlifespanで扱う場合、DB未設定時に確実に落ちないか。
