関数依存・DB調査レポート

調査日: 2026-05-10 / 対象: Backend/app, Backend/db, Backend/scripts, Backend/tests

調査ワークフロー に沿って、読み取り調査と静的解析を実施。書き込みは本ディレクトリ配下のみ。

結論

循環依存

アプリ内モジュールの循環importは検出なし。

ASTで app.* と config.* のimportグラフを抽出した範囲では閉路なし。関数呼び出しも主要フロー上の再帰・閉路は見当たらない。

依存構造

API層からCRUD/サービスへ直接依存する箇所が混在。

analysis はサービス経由が中心。dictionary はエンドポイントがCRUD、LLM、ベクトル化を直接束ねている。

DB現状

実質PostgreSQL + pgvector前提。

DATABASE_URL は汎用名だが、DDLとモデルは VECTOR(300), SERIAL, TIMESTAMPTZ, CREATE EXTENSION vector に依存。

Turso移行

難易度: 中。

CRUDの基本操作は移せるが、ベクトル型、DDL、接続ドライバ、SQLAlchemy型、日付型、リトライ方針の差し替えが必要。

主要依存図

矢印は「importまたは主要な実行時呼び出し」の向きです。core.database から app.api への依存はありません。main.py が app.api を読み込み、lifespanで条件付きに core.database を呼びます。

DBの現状

接続

config.config.get_database_url_env_key() が DATABASE_ENV_KEY を見て、未指定なら DATABASE_URL を使う。テストでは DEV_DATABASE_URL を強制。

モデル

Dictionary: id, term, description, meaning_vector, created_at, updated_at。ベクトルは pgvector.sqlalchemy.Vector(300)。

CRUD

全操作は TransactionManager.run 経由。競合時はSQLSTATE 40001 を文字列判定してリトライ。

DDL

Backend/db/dictionary.sql と reset_dictionary_db.py はPostgres DDL。CREATE EXTENSION vector が必要。

評価

観点良い点気になる点
可用性 DB未設定でも /analysis/vectorize などは動く。辞書参照はDB障害時にLLMへフォールバックする。 dictionary endpointはモジュール読込時の db = get_database() を保持するため、環境変数切替後の再初期化に弱い。
整合性 term にユニーク制約。bulkは既存語をスキップ。 meaning_vector はDB上NOT NULLだが、create_dictionary は None を受けられる。テストにもIntegrityError許容の痕跡がある。
拡張性 CRUDは1ファイルにまとまり、差し替え箇所は比較的少ない。 DB、LLM、ベクトル化の orchestration がendpoint/serviceに分散。services.dictionary のDB参照は未実装。
移植性 SQLAlchemyを使っているため、単純CRUDはDB差し替え余地がある。 pgvector.Vector とPostgres DDLに強く依存。CockroachDB向け依存もあるが実コードはPostgres互換前提。

Turso移行評価

Turso公式ドキュメントでは、Python/SQLAlchemy連携に sqlalchemy-libsql を使い、remote onlyでは sqlite+{TURSO_DATABASE_URL}?secure=true と auth_token を指定する例が示されています。ベクトルは拡張不要のネイティブ機能で、保存先はSQLite系の BLOB、投入時に vector32(...) などの関数を使う方式です。

現在Turso/libSQL側対応難度
create_engine(DATABASE_URL, connect_args={"application_name": ...}) sqlalchemy-libsql + sqlite+libsql / Turso URL + auth_token 接続設定を分岐。TURSO_DATABASE_URL, TURSO_AUTH_TOKEN を追加。 低〜中
id SERIAL PRIMARY KEY INTEGER PRIMARY KEY DDLとSQLAlchemy型をSQLite互換へ変更。 低
TIMESTAMPTZ DEFAULT NOW() SQLite互換の TEXT / DATETIME / CURRENT_TIMESTAMP UTC文字列保存かSQLAlchemy側defaultに寄せる。 低
pgvector.Vector(300), VECTOR(300) BLOB + vector32('[...]')、検索は vector_distance_cos 等 カスタムTypeDecoratorかCRUDで変換。既存APIは list[float] を返すため取り出し時に vector_extract が必要。 中〜高
CREATE EXTENSION vector 不要 初期化SQLから削除。 低
term.ilike("%q%") SQLite系は大文字小文字・日本語照合が異なる like か正規化列の追加を検討。 中
SQLSTATE 40001 リトライ SQLite/libSQL/Tursoの競合エラーは別体系 例外判定とリトライ方針を差し替え。 中

移行手順案

  1. DB抽象化を薄く追加Database にdialect判定を入れ、PostgresとTursoの接続生成を分ける。
  2. DDLを分岐Postgres用 dictionary.sql とTurso用DDLを分ける。Tursoは meaning_vector BLOB NOT NULL。
  3. ベクトル変換を隔離list[float] とDB保存形式の相互変換をCRUD層に閉じ込める。
  4. CRUDテストをSQLite/Turso互換で追加まずlocal libSQLまたはSQLite互換で create/read/list/update/delete を通す。
  5. 類似度検索を導入する場合だけTurso vector関数へ進む現状は完全な近傍検索を使っておらず、保存と取得が主用途。検索機能は後段でよい。

判断: 今すぐTursoに変える場合、辞書CRUDだけなら中難度。pgvector 相当のベクトル保存・抽出をきれいに保つなら中〜高難度。まず「DB保存形式をCRUD内へ隠す」小改修を入れてから移行すると安全です。

改善メモ

優先度 高

meaning_vector のNULL方針を統一。DBがNOT NULLなら create_dictionary で必ず300次元ベクトルを要求する。

優先度 高

refer_dictionary_v1 は未完成endpointから呼ばれているため、公開するなら同期/非同期バグとSSE出力形式を修正する。

優先度 中

dictionary endpointのCRUD/LLM/embedding調停をservice層へ寄せると、DB差し替え時の修正範囲が狭まる。

優先度 中

起動時・テスト時にDBインスタンスがキャッシュされる点を明示し、環境切替テストでは reset_database_instances を使う。
