# 用語スコア・会話テーマ（EMA）仕様書

会話中の用語に対する**素点＋バフ／デバフ**方式のスコアと、**会話テーマベクトル（EMA）**をバックエンドで計算し、フロントは結果のみ利用する方針の仕様である。

---

## 1. 目的とスコープ

| 項目 | 内容 |
|------|------|
| 目的 | バブル表示用のスコアとテーマ類似度の計算をサーバ側に集約し、再現性とチューニング単位を一本化する。 |
| 対象フェーズ（本書） | **現在実装済み〜近傍の改修**：テーマ／用語スコアの**計算ルール**、対応する **REST の形**、`term_score` / `score_building_blocks` の責務。ここまでを本書の主対象とする。 |
| スコープ（本文の記述） | テーマ EMA・スコア構成の要件、実ファイル／実 API／テスト方針。アルゴリズムの数値は「初期値」として計測で変更してよい。 |
| スコープ外（プロダクト全体の別ロードマップ） | **速度測定**、辞典 **DB のバッチ検索・重複排除**と **DB スキーマ再設計**、**複合語**・**embedding 基盤移行**（spaCy→別モデルなど）、**意味ベクトル／multi-sense 運用**の本番運用までの全配線、**LLM プロンプト**、**非同期レスポンス**（命中分を先に返す等）、**バッチ書き込み**。これらは本書が前提とする**後続の検討・実装**であり、別タスクとして進める。**本書の計算レイヤとは対立しない**（後述 §11 の「両立」を参照）。 |
| スコープ外（UI） | フロントの物理レイアウト（半径クリップ等）の最終調整のみをフロントに残すかは連携設計で別途決める。 |

## 2. 確定した設計判断（要約）

- **スコア**: 素点に対しバフ・デバフを適用する（単一の複雑な合成式に押し込めない）。
- **出現回数**: **回数軸**のグラフ（正規化／山型の加算。詳細は実装タスクで関数化）。
- **テーマベクトル**: **EMA**、更新係数 **α は固定**。
- **テーマ更新に含めない**: 無音に相当する空入力、**短い相槌のみ**の区間（ゲート判定は実装節参照）。
- **類似度の [−1,1] → 利用値**: **既定は線形** `(cosine_sim + 1) / 2`。非線形化は計測後のオプション。
- **外部コーパス**: **Wikipedia 系から事前計算した語→IDF のみ**。未ログ語は **語彙全体の平均 IDF**（定義は実装で固定）。
- **共起**: **PMI**（実装ではスムージング・正の部分のみ等を検討）。
- **1 文字除外**: **トークン表面の Unicode 長が 1** なら数字・記号・英字・和字を問わずスコア対象外（または強いデバフで実質除外。仕様上は「対象外」とする）。
- **品詞**: **名詞（広め）＋固有名詞**を対象とし、**代名詞のみ除外**（形態素エンジン依存のラベル一覧はコード／設定で固定）。

## 3. 実装前に数値または方針を置くべき項目（残課題）

以下は「未決定」ではなく、**初版で定数として置き、ログ・AB で調整する**想定である。

| ID | 項目 | 推奨の置き方 |
|----|------|----------------|
| R1 | **EMA の α** | 単一サーバ既定は `app/schemas/score_analysis.py` の `THEME_EMA_ALPHA_DEFAULT`（現状 `0.10`）。例レンジ `0.05`〜`0.15`。API 本体はチャンク長に依存しない。クライアントはリクエストで上書き可。 |
| R2 | **相槌除外** | 固定リスト（表層形）＋ **`content_token_count` しきい値**（例: 2 未満はテーマ更新スキップ）を併用。 |
| R3 | **テーマ状態の保持場所** | **サーバ**: セッション ID 付きで Redis／メモリ（単一インスタンス時のみ）等。**クライアント**: テーマベクトルを保持し毎回送信。運用要件で決定し、本仕様の API 形が従う。 |
| R4 | **IDF テーブルのキー** | 形態素の **`base_form` 推奨**（`surface` と揺れないようにする）。 |
| R5 | **PMI／共起** | `score_building_blocks` に隣接共起・PPMI の実装あり。**HTTP の** `POST /analysis/score/terms` **には共起バフは含めず**、`theme_linear` と `idf_scaled` と素点のみ。 |
| R6 | **PMI スムージング** | ゼロ回避の `+δ`（ディリクレ等）と、**負の PMI を切るか（PPMI）**を boolean で固定。 |
| R7 | **代名詞の POS ラベル** | GiNZa／Sudachi 等の実出力に合わせた **除外セット**を定数（現状 ``score_building_blocks`` のタプル）に列挙。必要なら後から YAML 等へ移行。 |

R3 は**本番環境での最適化論点**である。`/analysis/theme/*` とセッション付きテーマは**プロセス内メモリ**で実装済み。複数ワーカーや再起動で共有する必要があるときは **Redis 等への差し替え**とセッション寿命を別タスクで設計する。

## 4. 機能要件（スコア）

### 4.1 素点（例・確定順序は実装時にモジュール先頭で宣言）

- 回数軸による加点（ユーザー指定の「正規化グラフ」に基づく関数）。
- 素点に含めるかバフに含めるかは実装設計でよいが、**同一情報の二重計上を避ける**こと（例: 会話内頻度を素点に寄せ、Wikipedia は IDF のみ）。

### 4.2 バフ（例）

- 会話テーマベクトルとの **コサイン類似度**から算出した値（線形変換後、係数は定数）。
- **IDF**（事前テーブル、未ログは平均 IDF）。

現行 **`POST /analysis/score/terms`** の JSON 応答には上記のみ（共起 **PMI バフは載せない**）。`score_building_blocks` の PMI 関数はヘルパ・将来拡張用として残す。

### 4.3 デバフ／対象外（例）

- 表面長 1 のトークン: **処理しない**（スコア計算の入力から除外）。
- 名詞以外: **処理しない**。
- 代名詞: **処理しない**（名詞でも除外）。
- その他の減算ルールは別イテレーションで追加可能とし、本書では拡張点のみ予約。

## 5. 機能要件（テーマ EMA）

1. クライアントが **1 回送信するテキストチャンク**（現行と同様の `text`）を、**テーマ更新の 1 ステップ**の候補とする（既存 `POST /analysis/vectorize/sentence` の利用と整合）。
2. **相槌・無内容ゲート**を通過した場合のみ、`sentence_vector`（正規化済みを推奨）を `v_t` として EMA 更新する。
3. 更新式（概念）:  
   `theme' = normalize( (1 - α) * theme + α * v_t )`  
   初回は `theme = v_t` または零ベクトルからの初回のみ `theme = v_t` とするかは実装で明示。
4. **α は固定**（R1）。

## 6. API（現状実装と将来検討の区別）

### 6.1 現状実装（本書が整合を取る対象）

`app/api/endpoints/score_analysis.py` を `/analysis` にマウントしている（`app/api/__init__.py`）。

| メソッド・パス | 役割 |
|----------------|------|
| `POST /analysis/theme/chunk` | テーマ EMA を 1 チャンク分更新する。 |
| `POST /analysis/theme/session/reset` | セッションのテーマベクトルを破棄する。 |
| `POST /analysis/score/terms` | 複数語のスコア一括計算。 |

**スキーマ**: `app/schemas/score_analysis.py`  
**サービス実装**: `app/services/term_score.py`

辞典パイプラインや **DB 命中のみ先に返す**ような二段応答は、**本節の API の上に載る統合レイヤ**で後から足す（§11）。現行レスポンスが「処理完了まで一括」であることは、別ロードマップの応答見直しと矛盾しない（第一段／第二段のオーケストレーションとして切り離す）。

### 6.2 その他との関係（維持方針）

- `POST /analysis/vectorize`、`POST /analysis/vectorize/sentence`（既存）は**挙動互換を原則維持**し、テーマ更新は **`vectorize_sentence` をサービス経由で呼ぶ**形で共有する。
- 後続フェーズでフローを整理するとき、アップストリームの「辞書取得・非同期」が先行し、ダウンストリームは引き続き `POST /analysis/score/terms` に「確定済みベクトルを渡す」だけにできる。

## 11. プロダクト全体メモ（multi-sense・レイテンシ・非同期応答）との整合

会話重要語パイプラインの「途中まとめ」と、§1 で列挙した**後続ロードマップ**（速度測定、DB、フロー、複合語、embedding、意味ベクトル化、LLM、バッチ書き込み）は、**いずれも本書の条文と対立しない**ようにレイヤを分ける。

| 方針（全体） | 本仕様・実装での取り扱い |
|--------------|-------------------------|
| 文脈 embedding は **同一リクエスト単位で 1 回**に抑える | テーマ更新: チャンクあたり ``vectorize_sentence`` 1 回。用語スコア **バッチ**: チャンク全文の **隣接 bigram 用の形態素処理を 1 回**にし、各用語で再利用。辞典統合後も「統合ハンドラ」側で全文 embedding を増やさないこと。 |
| cosine 比較は軽い | ボトルネックは embedding／LLM／IO。比較はスコア本丸の構成要素でよい。 |
| multi-sense（曖昧語のみ複数 sense） | DB に sense 単位ベクトルを持つ想定。選択は ``best_sense_index_by_context``。現行 REST は単一 ``term_vector`` でよく、統合フェーズで拡張する。 |
| LLM とバッチ書き込み、**DB hit を先に返す**応答 | **オーケストレーション／辞典・統合 API の問題**。本書の `term_score`・`score_building_blocks` は「渡された確定済みベクトルとカウントで値を計算する」に徹する。第一段でヒットのみ返し miss を後続反映する構成は、この計算関数をそのまま第二段にも再利用できる。**「現在の同期的な REST」はロードマップの暫定形の一種**であり、恒久形を本書だけで拘束しない。 |

**矛盾のなかった整理**：旧稿で「すべて揃えてから一括返却」と読める記述がある場合は、「現行実装」の話とし、別ロードマップの「命中先行・非同期」を**上レイヤでの後付け可能**として扱う（§6.1 と上表）。
