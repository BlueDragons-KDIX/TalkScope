## 参加者

## 決定事項

| # | 決定内容 | 理由 |
|---|---|---|
| 1 | クリーンアーキテクチャで実装する | domain / application / infrastructure / presentation の4層分離。責務を明確にし、テストしやすくする |
| 2 | 状態管理に Zustand を採用する | App.tsx の神クラス化を防ぐ。ドメインごとにストアを分割し、不要な再描画を防ぐ。機能追加時の影響範囲を小さくする |
| 3 | デバッグ機能を `debug/` ディレクトリに完全分離する | デモテキスト・テストボタン・開発用UIを本番コードに混入させない |
| 4 | バブル重要度計算を DI（Strategy パターン）で実装する | 重要度アルゴリズムが今後変わることが確定している。`IImportanceStrategy` インターフェース経由で差し替え可能にする |
| 5 | レイアウト保存機能の口（`getLayout()` 関数）を用意する | 将来のカスタムレイアウト保存機能のために。サーバー連携は不要、フロントエンド内の関数として提供 |
| 6 | ウィンドウ種別を `IWindowDefinition` インターフェース + レジストリで拡張可能にする | 今後新種別を追加する予定がある。インターフェース実装 + レジストリ登録だけで追加できる設計にする |
| 7 | 文字起こし機能をメインシステムから完全切り離しする | 責務の分離。文字起こしはテキストを出力するだけ、重要語抽出以降は別システム |
| 8 | フロントエンド側の重要語抽出（`extractTerms`・`IT_TERMS`）を削除する | 将来フロー: 文字起こし → サーバー送信 → サーバーで抽出 → フロントへ反映。フロント側での抽出は不要 |
| 9 | フェーズシステム（発表中 / 発表後）を実装する | 発表終了後のサポート機能（議事録・重要語Top10等）を追加するため。各フェーズはレイアウト独立 |
| 10 | TDDで進める | 先にテストを書き、実装を合わせる |
| 11 | コミットメッセージは日本語 | 意図が伝わりやすい |
| 12 | pushは担当者が行う。AIは push しない | 担当者がレビュー後にマージ・pushする運用 |

## 未決事項・保留

- 発表後フェーズの具体的なウィンドウ構成（議事録以外は未定）
- Top10ウィンドウは発表中フェーズに置く可能性が高いが未確定
- 議事録はユーザーリクエスト時にサーバーで生成（詳細フロー未定）
