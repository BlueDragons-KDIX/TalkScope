# ADR-001: クリーンアーキテクチャの採用

## ステータス
承認

## コンテキスト
ハッカソンで一週間で作成したアプリをリファクタリングする。
現状の `App.tsx` はほぼすべての状態・ロジック・副作用を持つ神クラスになっており、
機能追加・テスト・保守が困難になっている。

## 決定内容
以下の4層クリーンアーキテクチャを採用する。

```
domain/         ← 外部依存なし。純粋な型・インターフェースのみ
application/    ← domain にのみ依存。ユースケース
infrastructure/ ← domain・application に依存。外部APIやストレージの具体実装
presentation/   ← すべての層を使用可。React UI
```

依存の向きは常に内側（domain）方向。infrastructure・presentation が domain に依存し、逆は禁止。

## 却下した選択肢

**現状維持（App.tsx中心）**: 状態が増えるにつれてさらに肥大化する。テストが書きにくい。却下。

**Feature-based 構造（機能別ディレクトリ）**: 小〜中規模では直感的だが、今回は機能追加が予定されており、責務の境界を明確にする方が長期的に有利。却下。

## 結果

- テスト可能性が向上する（domain・application は React 非依存でテストできる）
- 各層の責務が明確になり、新機能追加時の影響範囲が小さくなる
- 初期設計コストはかかるが、今後の機能追加・差し替えが容易になる
