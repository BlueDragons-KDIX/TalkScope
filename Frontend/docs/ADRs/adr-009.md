# ADR-009: Term の category / score（初級・中級・上級）の廃止方針

## ステータス
承認

## コンテキスト

`Term` には従来、技術領域を示す `category` と、難易度を示す `score`（旧 `level`、1=初級 / 2=中級 / 3=上級）があった。サーバー連携では重要語の意味説明と重要度ランキング（出現頻度・クリック）が主であり、フロント側の固定カテゴリ・難易度ラベルは仕様から外れた。

フィールドを即削除すると、モックデータ・IndexedDB・`addTerms` 呼び出し元で破綻しやすい。

## 決定内容

### 1. フィールドは型上残し、UI と算出から外す

| フィールド | 扱い |
|-----------|------|
| `category` | `string` として残す。`normalizeTermCategory()` は任意入力を常に `''` にする。`addTerms` 注入時に正規化。 |
| `score` | `number` として残す。初級/中級/上級の表示・`importanceRanking` の `term.score` 補正は廃止。 |

正規の難易度・領域情報は将来サーバー payload で別フィールドとして渡す想定とし、現行の `category` / `score` は互換用デッドフィールドとする。

### 2. 削除した UI・処理

- 詳細パネル・モーダルの難易度バッジ（初級/中級/上級）
- 文字起こし・バブルツールチップの `category` / `Score.{score}` 表示
- 重要度ランキング行の `Score.{term.score} / {category}` サブテキスト
- `defaultScoreStrategy` における `term.score` による加算（頻度 70% + クリック 30% のみ）

### 3. 重要度ランキングの「score」

`RankedTerm.score` はランキング用の合成値（頻度・クリック）であり、`Term.score`（難易度）とは別概念のまま維持する。

## 却下した選択肢

**`Term` から `category` / `score` を型ごと削除**: デモモック・DB スキーマ・既存テストの一括変更が大きく、段階的廃止の方が安全と判断。

## 結果

- 注入 API（`termStore.addTerms`）の形は維持しつつ、表示仕様をサーバー前提に寄せられる。
- `domain/entities/Term.ts` を正とし、`app/data/terms.ts` の `category` も `string` に揃えてビルド時の型二重化を解消する。
- モック `MOCK_IMPORTANT_TERMS` の `category` / `score` リテラルは削除せず、注入後に正規化または非表示となる。
